# Production Smoke Test Runbook (Phase 16 Cutover)

This runbook walks an operator through validating a freshly-deployed
Friendex instance against a staging guild before promoting it to a
production server. It is the human-facing companion to the machine-readable
checklist printed by `scripts/smoke_test_commands.py`.

The phase contract is recorded in
[`docs/04-migration-plan.md`](./04-migration-plan.md) §Phase 16.

## Pre-flight

Complete every item below **before** launching the bot. Each one is cheap
to verify and expensive to debug after the fact.

### 1. Environment variables

The runtime configuration is loaded from `.env` (or the process
environment) by `friendex.adapters.config.Settings`. Confirm the
following values exist:

| Variable | Required | Notes |
|---|---|---|
| `DISCORD_TOKEN` | **yes** | Bot token; placeholder values (empty string or `your_bot_token_here`) are rejected by the `validate_secrets` model validator. |
| `DEV_GUILD_ID` | optional | When set, the global command tree is also copied to that single guild and synced for instant availability. Production should leave this unset; the legacy `GUILD_ID` env name is still accepted as an alias. |

All other tunables (market hours, activity-tick K, hedge-fund APY, etc.)
default in `Settings`; override only when intentionally tuning.

### 2. Optional — migrator dry-run (legacy JSON cutover only)

If the deployment is migrating from the original JSON-backed bot
(`users.json`, `funds.json`, …), run the migrator in dry-run mode first
to surface any data issues without touching the target database:

```bash
uv run python -m friendex.adapters.persistence.migrate_json_to_sqlite \
  --guild-id <id> --dry-run --report
```

Honour the Phase 15b digest contract:

- `--guild-id` is **required** — the migrator never silently defaults a
  guild.
- `--dry-run` runs the migration end-to-end against a throwaway in-memory
  engine; the operator-supplied `--target` URL is **not opened, not
  schema-created, not written**.
- `--report` prints one line per migrated table to stdout, **sorted
  alphabetically by table name**, in the form `<table>: <count>`.
- The orphan-position check is **warn-not-fail** (emits a `WARNING` log
  line per orphan and continues; exit code stays 0).

Greenfield deployments with no JSON history can skip this section.

### 3. Database schema head check

Confirm the Alembic schema is up to date against the target database URL
(`Settings.database_url`, default `sqlite+aiosqlite:///data/friendex.db`):

```bash
uv run alembic upgrade head
```

This is idempotent — re-running on an already-current database is a
no-op.

### 4. Dependency sync

Lock the runtime dependency tree:

```bash
uv sync
```

This installs the production dependencies plus the dev group (test +
lint tooling). Re-running on a clean checkout is fast.

## Bot launch

With the pre-flight gates green:

1. Start the bot from the repo root:
   ```bash
   uv run friendex
   ```
2. Watch stdout for the structured **"ready"** log line — it confirms
   `on_ready` fired, the bot user id is bound, and the guild count is
   non-zero.
3. In your Discord client, type `/` in the staging guild's text channel
   and confirm the Friendex slash command tree is visible. If
   `DEV_GUILD_ID` was set, commands appear immediately; without it,
   global propagation can take up to ~1 hour after the first sync.

Do **not** proceed to the step-by-step verification until the "ready"
line is observed and the command tree is visible.

## Step-by-step verification

The checklist of slash commands, listener events, background tasks, and
startup/shutdown probes is generated by
[`scripts/smoke_test_commands.py`](../scripts/smoke_test_commands.py).

**Single source of truth:** this runbook does **not** duplicate the step
list. The script `scripts/smoke_test_commands.py` is the canonical list;
the runbook intentionally points at it so the two never drift. If a step
needs adjusting, change `STEPS` in the script (covered by
`tests/scripts/test_smoke_test_commands.py`) — never re-state the steps
inline here.

To print the checklist:

```bash
uv run python scripts/smoke_test_commands.py
```

The output is deterministic and byte-stable across runs (no timestamps,
no shuffling), so the operator can diff against a captured baseline if
desired. Walk every step in order against the staging guild and record
pass/fail in the sign-off table below.

## Post-flight

After working through every step:

1. **Graceful shutdown.** In the terminal running `uv run friendex`,
   send `Ctrl-C`. The signal triggers `bot.close()`, which drains the
   background task loops on their next iteration. No ERROR-level log
   lines should be emitted during teardown; the process should exit
   with status 0.
2. **Database file created.** Confirm the SQLite database exists at
   the configured `database_url` path (default
   `data/friendex.db`). For greenfield deployments the file should be
   present and non-empty after the first /buy or /daily.
3. **No ERROR-level log lines.** Grep the log capture for `"level":
   "error"` (JSON format) or `ERROR` (console format). The smoke test
   passes only if zero ERROR records were emitted across the run.
   Expected `DomainError` paths (e.g. `/daily` claimed twice, `/buy`
   against an opted-out user) surface as ephemeral user-facing messages
   via the Phase 13 error handler — those are correct behaviour and
   are logged at INFO/WARNING, not ERROR.
