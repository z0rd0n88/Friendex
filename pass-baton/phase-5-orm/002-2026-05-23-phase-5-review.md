## Findings

### MEDIUM — Decimal *quantisation* is not asserted; tests verify `==` value-equality only
- **Where:** `tests/adapters/persistence/test_orm.py` (every `assert <decimal> == Decimal('…')`).
- **Evidence (mutation test):** Replacing `DecimalText.impl = String` with
  `impl = Numeric` (the SQLite float-affinity storage the TypeDecorator exists to
  avoid) keeps **all 14 ORM tests green**, yet a direct probe shows quantisation is
  destroyed: with `Numeric`, `Decimal('100.00')` reloads with exponent `0`, not `-2`.
  The suite cannot distinguish the correct TEXT storage from the broken NUMERIC one,
  because `Decimal('100.0') == Decimal('100.00')` is `True` and none of the test
  fixtures use a float-inexact value. The product code is **correct** — a separate
  probe confirmed exponents `-2 / -1 / -3 / 0` all survive — but the test does not
  guard the invariant the baton claims it guards.
- **Fix:** In Phase 6 (or a quick test-only follow-up), add one assertion per Decimal
  field on the quantisation, e.g.
  `assert result.cash_balance.as_tuple().exponent == Decimal('9876.54').as_tuple().exponent`,
  and add a fixture value SQLite-float cannot hold exactly so a `Numeric` regression
  goes RED. The UTC counterpart is already strongly tested (see below) — bring Decimal
  to the same bar.

### LOW — Migration "no-drift" column check is tautological for this metadata-driven baseline
- **Where:** `tests/adapters/persistence/test_migrations.py::test_baseline_matches_orm_metadata`
  (the `_columns_by_table(migrated) == _columns_by_table(created)` assertion).
- **Evidence (mutation test):** Adding a spurious `Mapped[int]` column to `StockORM`
  leaves all 3 migration tests **green**. Both sides of the comparison derive from the
  *same* `Base.metadata` (`upgrade()` = `metadata.create_all`; the test's "created" DB =
  `Base.metadata.create_all`), so any ORM column change moves both sides together — the
  check is structurally incapable of detecting ORM↔migration drift for a baseline that
  *is* `create_all`. **Acceptable, not a bug:** drift is impossible by construction
  (baton 001's own argument), and the column-equality assertion only becomes load-bearing
  for future *hand-authored* incremental migrations in Phase 6+ (where it WILL catch real
  drift). The sibling table-set assertion against `_EXPECTED_TABLES` IS real — renaming a
  table to `trade_cooldowns_RENAMED` correctly fails 2 of the 3 tests.
- **Fix:** None required now. When Phase 6 introduces the first autogenerate migration,
  add a `compare_metadata`-based or explicit-DDL drift test against that migration so the
  column-level guarantee is exercised against independent DDL, not against `create_all`.

## Flagged-decision verdicts (judged, not rubber-stamped)

- **(a) `types.py` extraction (DecimalText / UtcDateTime) — ACCEPTABLE / sound.** It is
  the cohesive mechanism enforcing the two cross-cutting invariants, well under the size
  limit, single-responsibility, fully type-annotated (mypy clean), and reused by every
  ORM column. Extraction *reduces* hidden complexity vs. inlining into `orm.py`. Keep it.
- **(b) name-set equality vs `compare_metadata` — ACCEPTABLE with the LOW caveat above.**
  The rationale (SQLite collapses `TypeDecorator` columns to generic affinities, making
  `compare_metadata` report noisy type-only false diffs) is correct and verified. For a
  metadata-driven baseline the name-set + `_EXPECTED_TABLES` contract is adequate; the
  only real drift risk it misses (column-level) cannot occur for `create_all` and is
  deferred-appropriately to Phase 6's first incremental migration.

## What was verified positive

- **12/12 tables**, correct tablenames, `(guild_id, …)`-first composite PKs; child
  collections normalised to child tables (`voice_unique_channels`, `price_history`,
  `fund_investors`) — never blobs; `to_domain` aggregates take loaded children as args.
- **UTC round-trip is genuinely tested:** mutating `UtcDateTime.process_result_value` to
  strip tzinfo (`.replace(tzinfo=None)`) produces **8 RED tests** — the `.tzinfo is not
  None` + aware-`==` assertions catch it. Real, not tautological.
- **Alembic reversibility is real:** `upgrade head` creates exactly the 12 tables (+
  `alembic_version`), `downgrade base` drops them; renaming a table fails the suite.
- **Security:** all access via parameterized SQLAlchemy ORM — no string-built SQL; no
  hardcoded secrets; `DATABASE_URL` read from env with a safe loud-fail placeholder;
  `datetime.fromisoformat` parses only app-written values and raises on malformed input.
- **Dependencies:** zero added — `pyproject.toml` / `uv.lock` unchanged in the diff;
  `sqlalchemy[asyncio]>=2.0`, `aiosqlite>=0.20`, `alembic>=1.13` already declared.
- **SQLite FK enforcement off** (`PRAGMA foreign_keys = 0`): constraints are emitted but
  inert. Documented in both batons and explicitly deferred to Phase 6 — acceptable.
