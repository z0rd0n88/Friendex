## Hypothesis tree

- [x] **H1 (design):** `activity_price_tick` reads each user's `today` bucket
      via `IUserRepo.list_all`, gets their stock via `IPriceRepo.get`, computes
      `delta = compute_activity_return(bucket, k)`, proposed = current * (1 + delta/100),
      and clamps via `apply_floor_stall`. — *partially confirmed* by the
      Phase-4 digest: the domain function returns the *delta* (not a multiplier),
      and the original spec uses `current * (1 + ret_pct/100)`. We will follow
      the spec form for consistency with the original.
- [x] **H2 (design):** `inactivity_decay_tick` filters users where
      `now - last_activity > settings.inactivity_threshold_seconds` and applies
      `apply_inactivity_decay(current, settings.inactivity_decay, min_price)`.
- [x] **H3 (design):** `vc_boost_tick` boosts only users with an active session
      in `VoiceSessionStore`. Per original spec, each booster also needs
      per-user state (`VcExtraBoost` — ping_time/last_boost/end_time) to enforce
      the 1.5h window + 15min interval. Two store-design options:
      - **Option A (chosen):** Take `extra_boosts: list[VcExtraBoost]` as a
        method parameter and return the survivor list. Caller (Phase 9 task
        layer) owns the storage. Stays within the "no modify" constraint.
      - **Option B (deferred):** Add `VcExtraBoostStore` to
        `voice_session_store.py`. Out-of-scope for 8b (no modifies allowed).
- [x] **H4 (settings gap):** original spec uses a hard-coded `1.03` multiplier
      for the periodic VC boost. `Settings` does NOT have a `vc_extra_boost_multiplier`
      field. **Declared addition:** add `vc_extra_boost_multiplier: float = 1.03`
      to `src/friendex/adapters/config.py`. This is a tiny addition to the
      adapter layer and is required to keep the service free of magic numbers
      (per CLAUDE.md "no module-level constants for tunables"). Will note in
      digest. **Activity-tick K:** the original spec uses `ret_pct = ... ` whose
      formula collapses to `K · ln(1 + score)`. No K is exposed yet; will reuse
      `price_impact_k` is **wrong** semantically (trade impact ≠ activity K).
      Declared addition: `activity_tick_k: float = 0.5` (matches original
      spec's effective magnitude; see baton-runner digest-phase-4 §activity
      weighting).

## What's been ruled out

- Using `price_impact_k` for the activity-return K — semantically distinct
  (trade impact vs activity gain).
- Adding `VcExtraBoostStore` in this PR — contract forbids modifying
  `voice_session_store.py` or adding files outside the two named.
- Storing high_24h/low_24h on the tick path — Phase-3a correction 4 says
  these are dynamic from history; the spec explicitly omits
  `reset_24h_high_low`.
