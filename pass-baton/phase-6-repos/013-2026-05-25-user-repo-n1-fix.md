## Where things stand

The MEDIUM N+1 in `SqlUserRepository.list_all` / `list_active_in_last` (review
baton 006) is fixed, plus the adjacent LOW voice-channel ordering. Both list
methods now issue a **constant** number of queries regardless of user count: the
caller's single parent `SELECT`, then one batched `WHERE guild_id = :g AND
<owner/user>_id IN (:ids)` query per child table (long, short, bucket, voice),
grouped in memory and assembled from pre-grouped maps via the new
`_rebuild_many` / `_assemble` / `_bucket_from_maps` helpers. The single-user
`get` path is unchanged (still uses `_rebuild`, already O(1)). Voice channels now
load with `.order_by(VoiceUniqueChannelORM.channel_id)` so list order is
deterministic instead of rowid luck. Output is byte-identical to before:
longs/shorts keyed by `target_user_id`, today/week buckets with their
voice-channel lists, Decimal scale and UTC-aware datetimes preserved, `None`
buckets when absent. **Blocking state: none — gate is green, ready to commit.**
