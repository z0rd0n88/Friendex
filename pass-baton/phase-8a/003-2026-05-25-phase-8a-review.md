## Findings by severity

### HIGH — Lock key omits `guild_id`, violates ADR-0001 cross-guild isolation

ADR-0001 (`docs/adr/0001-per-guild-markets.md:72`) states verbatim: *"The
`LockManager` keys on `(guild_id, user_id)` so the same user in two guilds does
not serialise against themselves."* Phase-7 digest rule 5 reinforces this
(`baton-runner/br-2026-05-25-phase-7/digest-phase-7.md`): *"If a future phase
needs guild-scoped serialisation, key with a composite string
(e.g. `f"{guild_id}:{user_id}"`)."*

Both services lock on the **bare** `user_id`:
- `src/friendex/application/activity_service.py:112` (`_mutate`),
  `:258` (`_apply_stay_boost`), `:284` & `:297` (reset loops).
- `src/friendex/application/voice_ping_service.py:188` (`_apply_join_boost`),
  `:202` (`_credit`).

The Container (Phase 14, `docs/04-migration-plan.md:744`) builds **one**
`LockManager` and injects that single singleton into **every** per-guild
service scope. With bare-`user_id` keys, user "5001" in guild A and guild B
contend on the same `asyncio.Lock` — breaking the per-guild market isolation
ADR-0001 commits to. Not data corruption, but a documented architectural
invariant is violated, and the pattern will propagate to 8b–8f (TradingService,
FundService, etc.) where it compounds.

**Fix:** key every `locked(...)` call with the guild-composite id. Add a tiny
helper and route all six call sites through it, e.g.
```python
def _lock_key(self, user_id: str) -> str:
    return f"{self._guild_id}:{user_id}"
```
then `async with self._locks.locked(self._lock_key(user_id)):` everywhere
(reset loops use `self._lock_key(account.user_id)`). `self._guild_id` is already
stored on both services, so no signature changes are needed. Add a test that two
services with **different** `guild_id` but the **same** `LockManager` instance do
not serialise on a shared `user_id` (mirror
`test_different_users_do_not_block_each_other` with a barrier across two guild
scopes). Carry the composite-key convention into the 8b–8f digest.

### LOW — `VoiceSessionStore.link_ping` mutates a stored set in place

`src/friendex/application/voice_session_store.py:61` does
`session.from_ping_message_ids.add(message_id)`, mutating the stored
`VoiceSession`'s set in place. The project immutability rule prefers `replace`d
copies. Mitigating factors: this is **volatile, in-memory** state owned solely
by the store (never round-tripped through a repo, so no fake-vs-SQLite drift
risk), the store's own docstring documents it as volatile, and `link_ping` has
**no caller** in 8a (it will be wired in Phase 12). Still worth aligning with
the rule to avoid setting a "mutate volatile state in place" precedent. **Fix
(when wired):** rebuild the session via `replace(session,
from_ping_message_ids=session.from_ping_message_ids | {message_id})` and
`self._sessions[user_id] = ...` under the lock.

### LOW — `reward_voice_ping_response` read-modify-write is not atomic across awaits

`src/friendex/application/voice_ping_service.py:131-146` snapshots ping sessions
via `list_all()`, checks the cap / idempotency, then `_reward_for_session`
re-reads the same snapshot object and writes it back with `set()`. The
`VoicePingSessionStore` lock only guards each individual `get/set/pop`, not the
whole check-then-write span. Two responders processed concurrently could both
observe `len(first_10_joiners) < cap` and both be admitted past the cap (or a
late writer could clobber the other's append). Low likelihood today (no
concurrent caller until Phase 12 listeners) and not user-data corruption, but
the cap/idempotency guarantee is only as strong as the per-user `LockManager`
serialisation — which here keys on the *responder/host*, not the *ping session*.
**Fix (Phase 12 / when listeners drive it):** either serialise the
check-and-write under a store-level lock spanning the read-modify-write, or
lock on the ping `message_id` for the duration. Flagging now so 8b–8f author
keeps session-level RMW atomicity in mind.

## What is verified GOOD (do not regress)

- **A11 lock-leak fix** (`lock_manager.py:63-73`) is exactly correct: acquire
  loop moved inside `try`, `acquired` list tracks only held locks, `finally`
  releases `reversed(acquired)`. Test
  `test_cancel_mid_acquire_releases_already_held_locks` is genuinely
  load-bearing: pre-holds "b", cancels the victim while it holds "a" and parks
  on "b", then proves "a" is re-acquirable under a 1.0s timeout (a leak would
  hang). Reverting the fix fails this test.
- **A1–A6 / A7–A10** tests assert exact observable bucket/price values (e.g.
  150.00 = 100×1.50 for the stay boost; 120.00 for the join boost; today=0 /
  week=3 for reset; idempotency asserts `count("r1") == 1` plus unchanged
  points & price). Note the fakes return shared object refs, so the tests do
  **not** independently prove immutability — but the service code uses
  `dataclasses.replace` throughout for **persisted** aggregates (incl. copying
  `voice_unique_channels` list), so persisted-state immutability holds.
- **Lock discipline:** product code never touches `_locks`/`_ensure_lock`
  (only the test setup does, deliberately); `LockManager` is DI'd (constructor
  arg), never per-call; each critical section takes `locked()` once; no nesting
  of `locked()` for the same id; `handle_voice_leave` and `reward_*` chain
  *sequential* (not nested) lock acquisitions. Non-reentrancy respected.
- **Settings defaults** — all 10 match `docs/spec/original-skeleton.md`:
  `voice_ping_first_n_joiners=10` (spec `len(first_10) < 10`),
  `voice_ping_join_boost=1.20` (`*1.20`), `voice_stay_boost=1.50` (`*1.50`),
  `voice_stay_bonus_minutes=60.0` (`>= 60`), `voice_ping_base_points=5.0`,
  `voice_ping_fast/medium/slow_multiplier=3.0/2.0/1.0`,
  `voice_ping_host_credit=0.5` (`+= 0.5`), `photo_bonus_points=10.0`
  (`+= 10.0`). None contradict existing tunables.
- **guild_id as constructor arg:** consistent with ADR-0001 Approach C
  (`guild_id` captured once at scope construction; domain models stay
  guild-agnostic). Composes cleanly for 8b–8f — EXCEPT the lock-key gap above,
  which is exactly where the constructor `guild_id` must also be used.
- **Security:** no findings — internal service code, no secrets, no SQL string
  building (repos are parameterized in adapters), no raw user-input parsing,
  no sensitive logging.
