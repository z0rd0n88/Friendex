## Where things stand

Implementing Phase 9 background tasks under `src/friendex/adapters/tasks/`. Eight
task wrappers (activity tick, inactivity decay, liquidation, freeze check,
vc boost, daily reset, weekly reset, monthly rollover) plus a `BackgroundTask`
abstract base that owns the swallow-and-log error contract. All tasks are
guild-aware via a service factory injected at construction; the task layer never
re-derives per-guild keys (services own that ‚Äî Phase 8a digest).

Critical pinned conventions:
- `vc_boost_tick` storage-by-parameter (Phase 8b ¬ß5): task owns the
  `Iterable[VcExtraBoost]` storage and threads survivors tick-to-tick.
- `LiquidationTask` notifier injection (Phase 8f digest): task takes a
  `Callable[[LiquidationEvent], Awaitable[None]]`; **no `discord` import**.
- `accrue_apy` is retry-safe (Phase 8e digest): the day+hour gate plus 1-hour
  cadence ensures exactly-once per month without a new `SystemState` field.
- `activity_tick_k = 0.3` (Phase 8b calibration follow-up, already in config).

Dependencies: `freezegun >= 1.5` is already in `[dependency-groups].dev` of
`pyproject.toml` ‚Äî no new deps required.

## Per-guild fan-out design (TBD ‚Äî decide during implementation)

Two options, picking the cleanest one mid-implementation:
- **A)** Task takes a `service_factory: Callable[[str], TService]` and an
  `iter_guild_ids: Callable[[], Awaitable[Iterable[str]]]`. Each tick iterates
  guilds, builds a service for each, calls the method.
- **B)** Construct one task instance per guild at composition time; the task
  itself is single-guild.

A is favored ‚Äî single task instance, no instance-per-guild bookkeeping; the
service-factory pattern matches how Phase 14 will wire the LockManager
singleton across per-guild services. Tests cover N=2 guilds.

## Key design decision: cadence-as-declaration

Tasks declare cadence on `interval_minutes` / `interval_hours` class attrs;
the Phase 14 composition layer wraps `_run` in `discord.ext.tasks.loop(...)`
and binds `_loop`. This keeps the entire `adapters/tasks/` package free of
`discord` imports (per Phase 9 AC3 ‚Äî `LiquidationTask` must not import
`discord`; extended uniformly to all tasks for consistency). Tests check the
class attribute, not the loop.
