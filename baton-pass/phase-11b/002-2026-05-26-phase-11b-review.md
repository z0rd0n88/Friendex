## Findings

### 0 CRITICAL / 0 HIGH / 0 MEDIUM

### LOW

- **L1 — Inconsistent call style for `portfolio_snapshot`.**
  `AccountCog.balance` (Phase 11a, `account_cog.py:69`) calls
  `portfolio_snapshot(str(interaction.user.id))` positionally.
  `PortfolioCog.portfolio` (`portfolio_cog.py:79`) calls
  `portfolio_snapshot(user_id=str(target_user.id))` kw-style. Both work
  against the service signature `async def portfolio_snapshot(self, user_id: str)`,
  but the inconsistency is a small consistency snag for future readers.
  **Fix (non-blocking, optional, 11c-or-later):** standardise on kw-style
  (the service uses positional-or-keyword params, so either is legal, but
  kw is self-documenting). Leave for now — touching 11a is out of scope.

### INFO (notes, no action)

- **I1 — `AllowedMentions.none()` not applied in 11b cogs.** The Phase 10
  review (`baton-pass/phase-10/001-...`) flagged that embeds containing
  `<@user_id>` mentions need `allowed_mentions=AllowedMentions.none()` at
  the send call to suppress unintended pings. Phase 11a digest §4 carries
  this forward to Phase 11c `fund_cog` explicitly. `/trending` (public)
  and `/portfolio` (ephemeral) both render `<@user_id>` mentions via the
  embed builder. **Status:** discord.py does *not* ping from embed
  descriptions/fields by default — only from top-level `content=` — so this
  is a defence-in-depth nit, not a behaviour bug today. The carry-forward
  to 11c remains the right place to land the project-wide hardening; do
  not surgically patch 11b alone.
- **I2 — Local re-import of `discord.app_commands` inside test functions.**
  `test_portfolio_cog.py:174`, `test_stats_cog.py:336,348` each
  `import discord.app_commands as app_commands` inline despite `discord`
  being imported at module level. This mirrors the established pattern in
  `test_account_cog.py:220` (Phase 11a). Pre-existing convention; not
  worth changing.
