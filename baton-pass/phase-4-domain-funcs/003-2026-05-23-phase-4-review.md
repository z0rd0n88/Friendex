### MEDIUM

- **MED-1 — `compute_net_worth` justification is factually wrong (doc only; math
  is correct).** `src/friendex/domain/fund_math.py:17-27` (docstring) and both
  implementer batons (`001` §net-worth, `002` "Net-worth valuation decision")
  state the original spec leaves `calculate_net_worth` *undefined*. It is in fact
  **defined** at `docs/spec/original-skeleton.md:320-338`. The original computes a
  short's contribution as `shares*entry_price - shares*current_price`; the
  implementation uses `locked_cash + locked_fund - shares*current_price`. These
  are **numerically equivalent for every reachable state** because the original
  maintains the invariant `locked_cash + locked_fund == shares * entry_price`
  (collateral locked at open == notional; proportional release on partial cover
  preserves it — verified against spec lines 1175-1201 and 1266-1283). So there
  is **no arithmetic or sign error** and the result is correct. *Fix:* correct the
  docstring + a follow-up note — reference spec:320, state the formula is an
  equivalent collateral-based decomposition of the spec's
  `entry_value - current_value`, and record that the equivalence **depends on**
  the `locked == shares*entry_price` invariant, so the Phase-7/8 short service
  must preserve it (otherwise `compute_net_worth` and the original diverge).

- **MED-2 — `apply_floor_stall` attenuation magnitude is unpinned (test gap).**
  `tests/domain/test_price_engine.py:185-200`
  (`test_floor_stall_attenuates_down_move_near_floor`) asserts only the relative
  inequality `far_drop > near_drop`, and its `near` case (current 75 → proposed
  65) is dominated by the floor clamp, so changing `_ATTENUATION_DISTANCE`
  (`src/friendex/domain/price_engine.py:42`) from `10.0` to `1000.0` — or
  removing the attenuation term — passes every test. The attenuation curve (the
  one piece of `apply_floor_stall` that mirrors the original spec, lines 485-487)
  could regress silently. *Fix:* add an exact-value assertion away from the floor,
  e.g. current=78.00, proposed=70.00, min=70.00 → distance 8, atten min(1,0.8)=0.8
  → expected `78.00 - 8*0.8 = 71.60`; this pins both the `/10.0` window and the
  formula.

### LOW

- **LOW-1 — `get_engagement_tier` assumes `score in all_scores` and uses
  `list.index` (first match).** `src/friendex/domain/activity.py:104`. With
  duplicate scores, `index()` returns the *first* (best-ranked) position, so tied
  members all get the top tie's tier — a defensible but undocumented choice; and a
  `score` not present in `all_scores` raises `ValueError`. The docstring states the
  precondition, and no current caller violates it, so non-blocking. *Fix (optional):*
  when Phase-9/stats wires this up, either compute rank by `sum(s > score for s in
  all_scores)` (count-greater, tie-stable) or document the tie semantics at the call
  site.

- **LOW-2 — `_ATTENUATION_DISTANCE` / `_MIN_DISTANCE` are module constants, not
  `Settings`.** `src/friendex/domain/price_engine.py:42-43`. Consistent with the
  activity weights (already flagged as a deferred follow-up in baton 001) and with
  the original monolith hardcoding them; acceptable for now. *Fix (optional):*
  promote to `Settings` in the same future pass that promotes the activity weights.
