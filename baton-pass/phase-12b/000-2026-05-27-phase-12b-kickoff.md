## Method note: `record_message` signature

`ActivityService.record_message` (existing in
`src/friendex/application/activity_service.py:164`) is actually
`(author_id, has_attachment, is_reply, channel_id)`. STATE.md's
shorthand ("user_id, has_media, is_reply, mention_count") is a paraphrase —
the listener calls with the real signature: `author_id=str(author.id)`,
`has_attachment=bool(message.attachments)`,
`is_reply=message.reference is not None`,
`channel_id=int(message.channel.id)`. `mention_count` is not in the service
contract — replies are detected via `message.reference`, role-pings via
`message.role_mentions` against `settings.vc_ping_role_ids` (the
spec's `is_voice_ping_message` rule).

## CF-4 boost-list source decision

The voice-ping service maintains `VoicePingSession.extra_joiners` per ping
session (the responders beyond the first-N cap). The cleanest pull point
for the listener-driven seed is **`VoicePingService.collect_extra_boosts()`**
— a new tiny query that walks open ping sessions for the guild and emits
one `VcExtraBoost` per `extra_joiner`. The listener calls it after every
join/switch and pushes the result into
`VcBoostTask.set_store_for_guild(guild_id, …)`. This keeps the listener
free of session-store internals.

## VC ping rule decision (B1)

Per `docs/spec/original-skeleton.md:494-503`:
> A message counts as a voice ping if it mentions any VC ping role.

→ `member.voice and member.voice.channel` (author is in a VC) **AND**
`any(r.id in settings.vc_ping_role_ids for r in message.role_mentions)`.
Default `vc_ping_role_ids = []` (per `adapters/config.py:120`), so without
operator config no message triggers the path — exact replica of the spec.
