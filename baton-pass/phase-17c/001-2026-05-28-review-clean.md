## Security review (DM send + embed escaping)

- **DM send safety.** `interaction.user.send(embed=..., allowed_mentions=
  AllowedMentions.none())` — Discord's DM API; no user-supplied URL,
  no attachment, no `file=`. `build_intro_embed` is static (no
  user-controlled fields). Safe.
- **Race conditions on `intro_shown`.** The flip happens under the
  composite-key lock (`self._lock_key(user_id)` resolves to
  `f"{guild_id}:{user_id}"`). LockManager is non-reentrant; the
  read-then-write happens inside the lock. Two concurrent `/optin`
  invocations serialise — only the first sees `intro_shown=False`,
  the second sees `True`. No double-DM.
- **`AllowedMentions.none()` correctness.** Verified on every new
  send. The `<@{user.id}>` mention in the invest ack does NOT ping —
  `AllowedMentions.none()` zeroes `roles`, `users`, `everyone`.
- **Embed-content escaping for fund/user names.** The invest ack
  description format string interpolates `user.id` (numeric Discord
  snowflake — not user-controlled text) and `decimal_amount`
  (Decimal). No display-name or fund-name string interpolation, so
  no mention-injection or markdown-escape risk.
