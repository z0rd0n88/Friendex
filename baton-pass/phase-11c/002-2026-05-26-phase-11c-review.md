## Findings

### LOW-1 — Money-mutation test values don't expose IEEE-754 drift

`tests/adapters/discord_bot/cogs/test_fund_cog.py:280, 336` use amount
literals `100.50` and `75.25`. Both are exactly representable in binary
floating-point, so swapping `Decimal(str(amount))` for `Decimal(amount)`
in the cog leaves `Decimal("100.50") == Decimal(100.50)` and the
`assert_awaited_once_with(..., Decimal("75.25"))` / `args[1] == Decimal("100.50")`
assertions still pass. The brief anticipated this. Fix is to add at least
one test using a non-dyadic literal — e.g. `0.1` or `33.33` — where
`Decimal(0.1) → Decimal('0.10000000000000000555…')` and
`Decimal(str(0.1)) → Decimal('0.1')` diverge. Concrete fix:

```python
# in test_fund_withdraw_calls_service_with_decimal_and_utc_now
await FundGroup.withdraw.callback(group, interaction, amount=0.1)
fund_service.withdraw.assert_awaited_once()
args, _kwargs = fund_service.withdraw.await_args
assert args[1] == Decimal("0.1")  # NOT Decimal(0.1) — would catch the drift
```

Not a blocker — the docstring on `withdraw` (`fund_cog.py:173-176`)
spells out the invariant; a future hardening pass can land the
non-dyadic literal alongside.

### LOW-2 — `/fund info` APY assertion is weakly load-bearing

`test_fund_info_passes_base_and_effective_apy_to_builder` only asserts
`"15.00%" in rendered`. With `penalty=None`, `compute_effective_apy`
returns `base_apy` unchanged, so the rendered effective APY equals the
base. A regression that hardcoded `effective_apy=0.15` (instead of
calling `compute_effective_apy`) would slip past this test, as would
flipping the `base_apy=` / `effective_apy=` builder kwargs. Concrete
fix: add a test that passes a `FundPenalty`-bearing Settings (or
monkey-patch `compute_effective_apy` so the two APYs diverge in the
rendered embed and the test can assert both `15.00%` AND a distinct
effective APY value). Per the cog's open question (its own baton lists
this as "FundPenalty surfacing"), this is a Phase-13 enhancement
candidate, not a Phase-11c fix.

### LOW-3 — ARCH.md is stale post-commit

`python3 .githooks/gen_arch.py` produces a 46-line diff (the two new
cog files are not yet listed in the file-tree map). The pre-commit hook
didn't run because `core.hooksPath` is not configured in the worktree
(CLAUDE.md notes the user must do this once via
`git config core.hooksPath .githooks`). Concrete fix: run
`python3 .githooks/gen_arch.py` and either include the regenerated
ARCH.md in the phase-11c commit (amend or follow-up commit) or rely on
the next phase commit to refresh it. Not a Phase-11c product defect.
