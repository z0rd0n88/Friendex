### LOW

- **L1. Test gap: in-place dict aliasing on `fund.investors`.**
  `tests/application/test_fund_service.py:706-758` ‚Äî the happy-path and
  accumulation tests assert the final `fund.investors` value after
  `invest`, but only one round-trip through the fake repo. If the
  implementation were to `fund.investors[investor_id] = ‚Ä¶` directly on
  the snapshot returned by `IFundRepo.get` (without `dict(...)`), the
  fake-repo's stored dict object is the same reference, so the assertion
  still passes (M5 mutation confirmed this). The current code does clone
  defensively (lines 447), but the safety is unenforced by tests. A real
  SQL repo will detach the dict on read, so the production blast radius
  is small ‚Äî but a future fake-repo that returns a fresh copy per `get`
  call would expose the bug and the test would still pass.
  **Fix:** add a regression test that asserts the dict identity is
  fresh, e.g. `seeded_investors = {}; seed fund with investors=seeded_investors;
  await service.invest(...); assert seeded_investors == {}` (the seeded
  dict, kept by the test, must remain unmutated). Cheap, two lines.
  Defer to 17c if the team prefers ‚Äî not blocking.

### INFO

- **I1. Cog ack carry-forward (expected ‚Äî do not flag as a 17b miss).**
  `fund_cog.py:237-265` ‚Äî `/fund invest` calls
  `await fund_service.invest(...)` and returns silently. Discord requires
  every interaction to be acknowledged within 3s, so a successful invest
  surfaces as "this interaction failed" in the client. The work prompt
  explicitly excluded cog Python changes from 17b and the baton 000 NOTES
  flags this as a 17c task. Pinned in the phase-17b digest below.

- **I2. Opt-in check decision.** `invest` does **not** consult
  `_check_opt_in` for the target fund's manager. This is acceptable for
  17b ‚Äî investing is a financial transaction, not a trade; the spec did
  not require the gate; the 17a `_check_opt_in` helper is a per-stock
  toggle, not a per-fund one. No action required ‚Äî recording the
  decision so 17c does not invent an unjustified opt-in requirement.

- **I3. Manager-balance accounting drift on `_quantise`.** In
  `accrue_apy` (lines 364-388), `manager_balance = fresh.cash_balance ‚àí
  sum(fresh.investors.values(), Decimal("0.00"))` is *not* quantised
  before passing to `compute_apy_accrual`. If repo data ever carries
  more than two decimal places (it shouldn't, the persistence layer
  quantises on write), the accrual computation would carry that drift.
  Not a real issue today because every other site quantises, but worth
  recording as a soft invariant.
