## Security review (ecc-security-review-style scan)

Log-payload sensitivity check (the only new product surface):

- **Event name:** ``"account.optin_intro_dm_forbidden"`` — stable, dotted
  namespace, no user-provided substring; safe.
- **Payload keys:** ``user_id`` (``str(interaction.user.id)``), ``guild_id``
  (``guild_id_of(interaction)``). Both are Discord snowflake IDs cast to
  string — non-secret, non-PII identifiers that the bot already logs
  elsewhere (see ``error_handler.py`` and ``activity_service.py`` logs).
- **NOT logged:** username, display name, intro embed contents,
  confirmation embed contents, channel ID, message ID, token, anything
  user-typed.
- **No log injection risk:** the only string field is ``"%s" % event_name``
  in the underlying handler; ``extra`` dict goes through ``LogRecord``
  attribute setattr, not format-string substitution.
- **Level choice:** ``INFO`` is appropriate — this is an operator-visibility
  signal (DM block rate), not user error and not a security event.
  Per-incident frequency is bounded by the rate of first-time ``/optin``
  invocations per server.
- **Concurrency:** stdlib ``logging`` is thread-safe; no new shared state.

Zero security findings.
